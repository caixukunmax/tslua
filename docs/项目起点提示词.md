2. 项目背景
我希望构建一套名为 “TS-Skynet Hybrid” 的开发框架。
核心痛点：Skynet 性能极佳，但 Lua 开发缺乏类型安全和现代工程化能力；TypeScript 开发体验好，但直接运行在 Node.js 中缺乏 Skynet 的 Actor 模型。
核心目标：使用 TypeScript 编写核心业务逻辑，通过编译转换为 Lua 在 Skynet 中运行，同时支持在 Node.js 环境下直接运行用于逻辑验证和单元测试。

3. 核心技术栈
开发语言: TypeScript 5.x
编译工具: TypeScriptToLua (TSTL)
运行环境 A (开发/单机): Node.js
运行环境 B (生产/集群): Skynet (C/Lua)
构建工具: Vite 或 Webpack (可选，建议提及)

4.1 抽象接口层

这是框架的灵魂。TS 业务代码必须不直接依赖 Node.js 的 fs、net 模块，也不直接依赖 Skynet 的 skynet.* API。
需要定义一套标准接口（例如 ILogger, INetwork, ITimer, IDatabase）。
4.2 双模式运行时适配

Node.js 适配器: 实现上述接口，底层使用 Node.js 原生库（如 console.log, setTimeout）。用于本地跑通逻辑、单机测试。
Skynet 适配器: 实现上述接口，底层封装 Skynet 的 Lua API（如 skynet.error, skynet.start, skynet.call）。用于生产环境高性能运行。
4.3 异步模型统一

需求: 开发者在 TS 中统一使用 async/await。
Node.js 侧: 映射为标准的 Promise。
Skynet 侧: 这是关键难点。需要将 TS 的 Promise 转换为 Lua 的 coroutine。当业务代码 await 一个 RPC 调用时，在 Lua 层必须正确地 yield，并在回调中 resume，绝不能阻塞 Skynet 线程。
4.4 编译与构建流程

设计 TypeScript 到 Lua 的转换规则。
解决 OOP 映射：如何将 TS 的 class 优雅地映射为 Lua 的 table + metatable 结构，以保持性能。
解决模块依赖：TS 的 import/export 如何转换为 Skynet 的 require 机制。
4.5 调试与错误追踪

设计方案支持 SourceMap，使得 Skynet 报错（Lua 行号）能映射回 TS 源码。
在 Node.js 模式下支持 VS Code 原生断点调试。
